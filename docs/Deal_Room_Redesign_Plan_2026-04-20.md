## 1. 왜 지금 바꿔야 하는가 — 현 상태 진단

### 1.1 한 건의 거래가 오가는 화면 수 (현재)

```
/exchange            (매물 목록)
  ↓ 카드 클릭
/exchange/[id]       (L0/L1 매물 상세)
  ↓ NDA 버튼
/exchange/[id]/nda   (NDA 서명)
  ↓ 딜룸으로
/deals/[id]          (딜룸 본체 — 채팅·문서)
  ↓ 필요 시 복귀
/exchange/[id]/dataroom   (실사 자료)
/exchange/[id]/loi        (LOI 제출)
/exchange/[id]/escrow     (에스크로)
/exchange/[id]/flow       (플로우 대시보드)
```

거래 1건에 **최대 8개 URL** · 공통 헤더/요약이 각 화면마다 중복 · 사용자가 "지금 내가 뭘 해야 하는지" 화면마다 다시 찾아야 함.

### 1.2 딜룸 자체의 문제

- 현재 TopBar `딜룸` → `/deals` 는 **대시보드** (KPI 카드·차트·필터) 가 먼저 보이고, 실제 협상하려면 다시 한 번 클릭해야 함.
- 대시보드의 시각 요소(도넛 차트·파이프라인 막대)는 *매니저* 관점 데이터이지, *딜 당사자*의 "지금 수락/거절" 의사결정에 기여 0.
- 결과: **핵심(딜) 대신 보조(대시보드)가 기본**. 사용자 말씀 그대로.

### 1.3 이 서비스의 단일 지표

> **Deal Velocity = 관심 표시 → 체결 평균 소요 일수.**

이 지표를 **-30%** 당기는 것이 이번 재설계의 유일한 목표. 그 외 (시각적 완성도·대시보드 풍부함·차트 다양성)는 전부 후순위.

## 2. 핵심 설계 원칙 — "4 Rules"

| # | 원칙 | 구체화 |
|---|------|--------|
| 1 | **하나의 딜 = 하나의 URL** | `/deals/[id]` 만 존재. `/exchange/[id]/*` 는 모두 `/deals/[id]` 로 301 redirect. |
| 2 | **진행 단계 ≠ 페이지 전환** | NDA·LOI·실사 요청·서명은 **모달**로. 티어 업그레이드 시 URL 유지, 딜룸 내부 상태만 변경. |
| 3 | **1 화면 = 1 Primary CTA** | 티어 상태에 따라 자동 결정되는 *하나의* 큰 버튼. "지금 뭘 해야 하나" 에 고민이 없게. |
| 4 | **협상은 상시 우측 고정** | 채팅 + 문서는 항상 열려 있음. 메시지 하나 보내려고 탭 이동·스크롤 필요 없음. |

## 3. 신 URL 구조

### 3.1 Before / After

| Before | After |
|---|---|
| `/exchange/[id]` | `/deals/[id]` (L0 진입) |
| `/exchange/[id]/nda` | `/deals/[id]` + `?action=nda` 모달 |
| `/exchange/[id]/dataroom` | `/deals/[id]#dataroom` 섹션 스크롤 |
| `/exchange/[id]/loi` | `/deals/[id]` + `?action=loi` 모달 |
| `/exchange/[id]/escrow` | `/deals/[id]#settlement` 섹션 |
| `/exchange/[id]/flow` | `/deals/[id]` 상단 Stepper 에 이미 표시 → 삭제 |
| `/deals` (대시보드) | `/deals` = **활성 딜 1개면 자동 리다이렉트**, 여러 개면 카드 리스트 (KPI 제거) |

### 3.2 URL 당 기대 진입·이탈 흐름

```
매물 탐색(/exchange)
  ↓ 카드 "딜룸 열기"
/deals/[id]   ← 모든 거래 행동이 여기서 일어남
  ↓ 체결 완료
/my/deals     ← 완료 아카이브 (선택 이동)
```

중간에 `/exchange/[id]/*` 가 절대 등장하지 않음.

### 4.4 매물 사진 갤러리 (Media Hero)

딜룸 최상단(스텝퍼 위)에 상시 노출되는 사진 영역. 거래 당사자가 "무엇을 사고파는지" 를 초단위로 체감할 수 있어야 한다.

**구성:**
- **Desktop**: 5-cell 그리드 (대표 사진 1 × 썸네일 4) · 높이 380px · 1차 딜 identity 확보
- **Tablet (≤900px)**: 2×2 그리드 · 높이 280px
- **Mobile (≤560px)**: 단일 대표 이미지 · 높이 220px · 좌우 스와이프

**상호작용:**
- 대표/썸네일 어느 것을 클릭해도 `galleryModal` (라이트박스) 오픈
- 라이트박스 = 1 primary + 12 thumbnail strip · Esc 로 닫기
- `"사진 12개 전체 보기"` CTA 로도 진입 가능
- 이미지 없는 딜은 gradient placeholder 5 variants 로 fallback (등급 뱃지·주소 텍스트 오버레이로 identity 유지)

**데이터 소스:** 기존 `listings.images[]` 재사용. Supabase Storage 공개 URL. 신규 API 불필요.

### 4.5 Data Room · PDF 뷰어 / 다운로드

중앙 영역 하단에 **Data Room 카드** 를 상주시켜 문서를 즉시 확인·다운로드 가능하게 한다. 실사 자료 요청 → 수신 → 열람 → 다운로드 사이클이 *한 페이지 안에서* 완결.

**문서 목록 행 구조 (티어별 게이팅):**

| 티어 | 노출 문서 | 액션 |
|------|-----------|------|
| L0 | 요약본 (이름만) · 파일 자체는 잠김 | 🔒 `NDA 서명 후 열람` |
| L2 | NDA 체결 이후: 감정평가서 · 등기부등본 · 권리분석 | 👁 미리보기 · ↓ 다운로드 |
| L3 | + 임대차계약서 · 실사보고서 초안 | 👁 · ↓ · 💬 (이 문서에 질문) |
| L4 | + 전자계약서 · 에스크로 이체증 | 👁 · ↓ · ✍ 전자서명 |

**미리보기 (`pdfModal`):**
- 인라인 렌더링 (PDF.js 또는 html 기반 템플릿 — 외부 탭 이동 0)
- 상단 툴바: 파일명 · 페이지 카운터 · **[↓ 다운로드]** 버튼 · Esc 로 닫기
- 모바일에서는 전체화면 스위프 뷰 (pinch-zoom 허용)
- 다운로드 = 서버 서명 URL (`/api/v1/files/:id/signed-url`, 15분 만료) → 감사 로그 자동 기록

**데이터 소스:** 기존 `deal_documents` 테이블 + `documents.bucket_path` 재사용. 기존 `/api/v1/deals/[id]/documents` 확장 (썸네일·page_count 컬럼만 추가).

- 문서 (`guide/[topic]`) 에서 구 경로 링크 업데이트
